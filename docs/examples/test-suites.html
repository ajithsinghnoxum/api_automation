Users API 21 tests

The Users endpoint returns 10 users with nested objects (address, company, geo). These tests demonstrate the widest range of validation types.

API Endpoints Used

MethodEndpointDescription
GET/usersReturns all 10 users
GET/users/1Returns a single user
GET/users?username=BretFilter by query param
GET/users/9999Returns 404

Tests Included

1. Return all users

Validates the response is an array of exactly 10 users.

{
  "name": "should return all users",
  "method": "GET",
  "endpoint": "users",
  "expectedStatus": 200,
  "validations": [
    { "type": "isArray" },
    { "type": "arrayLength", "exact": 10 }
  ]
}

What each validation does:

  • isArray — Checks that the entire response body is a JSON array (not an object or primitive)
  • arrayLength exact: 10 — Asserts the array has exactly 10 items (JSONPlaceholder always returns 10 users)

2. Return user by ID

Checks specific field values with equals and exists.

{
  "name": "should return user by ID",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "equals", "path": "id", "value": 1 },
    { "type": "equals", "path": "name", "value": "Leanne Graham" },
    { "type": "exists", "path": "email" },
    { "type": "exists", "path": "address" },
    { "type": "exists", "path": "company" }
  ]
}

What each validation does:

  • equals path:"id" value:1 — Checks that response.id is exactly 1 (strict equality)
  • equals path:"name" value:"Leanne Graham" — Verifies the user's name matches the expected string exactly
  • exists path:"email" — Confirms the email field is present in the response (not null/undefined)
  • exists path:"address" — Confirms the address nested object exists
  • exists path:"company" — Confirms the company nested object exists

3. Validate user schema

Uses schema to validate types of multiple fields at once.

{
  "name": "should validate user schema",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    {
      "type": "schema",
      "properties": {
        "id": "number",
        "name": "string",
        "username": "string",
        "email": "string",
        "phone": "string",
        "website": "string"
      }
    }
  ]
}

What each validation does:

  • schema — Validates multiple fields' types in one check: id must be a number, name/username/email/phone/website must be strings. Fails if any field has the wrong type.

4. Validate nested objects

Uses hasProperty and nested schema to check the address and geo objects.

{
  "name": "should validate nested address object",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "hasProperty", "path": "address", "property": "street" },
    { "type": "hasProperty", "path": "address", "property": "city" },
    { "type": "hasProperty", "path": "address", "property": "zipcode" },
    {
      "type": "schema",
      "path": "address.geo",
      "properties": { "lat": "string", "lng": "string" }
    }
  ]
}

What each validation does:

  • hasProperty path:"address" property:"street" — Checks that the address object has a street key
  • hasProperty path:"address" property:"city" — Checks that the address object has a city key
  • hasProperty path:"address" property:"zipcode" — Checks that the address object has a zipcode key
  • schema path:"address.geo" — Validates the nested address.geo object has lat (string) and lng (string)

5. Regex email validation

Uses regex to validate email format.

{ "type": "regex", "path": "email", "pattern": "^[^\\s@]+@[^\\s@]+\\.[^\\s@]+$" }

What each validation does:

  • regex path:"email" pattern:"^[^\\s@]+@..." — Tests the email field against a regex pattern that matches standard email format (no spaces, has @, has domain with dot)

6. Query parameter filtering

Filters users by username=Bret and validates every result matches.

{
  "name": "should filter users by query param",
  "method": "GET",
  "endpoint": "users",
  "queryParams": { "username": "Bret" },
  "expectedStatus": 200,
  "validations": [
    { "type": "isArray" },
    { "type": "arrayLength", "min": 1 },
    {
      "type": "arrayEvery",
      "validations": [
        { "type": "equals", "path": "username", "value": "Bret" }
      ]
    }
  ]
}

What each validation does:

  • isArray — Confirms the filtered response is still an array
  • arrayLength min: 1 — Ensures at least 1 user matched the filter (not an empty array)
  • arrayEvery > equals path:"username" value:"Bret" — Iterates over EVERY item in the array and asserts each one has username equal to "Bret" — guarantees no non-matching results leaked through

7. arrayEvery — all users have required fields

{
  "type": "arrayEvery",
  "validations": [
    { "type": "exists", "path": "id" },
    { "type": "typeOf", "path": "name", "expected": "string" },
    { "type": "typeOf", "path": "email", "expected": "string" },
    { "type": "exists", "path": "address" }
  ]
}

What each validation does:

  • arrayEvery — Runs the nested validations against EVERY item in the users array
  • exists path:"id" — Each user must have an id field
  • typeOf path:"name" expected:"string" — Each user's name must be a string (not null, number, etc.)
  • typeOf path:"email" expected:"string" — Each user's email must be a string
  • exists path:"address" — Each user must have an address object

8. arrayFind — find user by name

Finds "Ervin Howell" regardless of array position and validates associated fields.

{
  "type": "arrayFind",
  "where": { "path": "name", "value": "Ervin Howell" },
  "validations": [
    { "type": "equals", "path": "id", "value": 2 },
    { "type": "equals", "path": "username", "value": "Antonette" },
    { "type": "contains", "path": "email", "value": "@" }
  ]
}

What each validation does:

  • arrayFind where:{path:"name", value:"Ervin Howell"} — Searches the array for the first item where name equals "Ervin Howell", then runs nested validations on THAT item only
  • equals path:"id" value:2 — The found user's id must be 2
  • equals path:"username" value:"Antonette" — The found user's username must be "Antonette"
  • contains path:"email" value:"@" — The found user's email must contain an @ symbol (substring check)

9. arrayNone — no empty names

{
  "type": "arrayNone",
  "validations": [
    { "type": "equals", "path": "name", "value": "" }
  ]
}

What each validation does:

  • arrayNone > equals path:"name" value:"" — Asserts that NO item in the array has an empty string name — if even one matches, the validation fails

10. arraySome — at least one .org website

{
  "type": "arraySome",
  "validations": [
    { "type": "regex", "path": "website", "pattern": "\\.org$" }
  ]
}

What each validation does:

  • arraySome > regex path:"website" pattern:"\\.org$" — Asserts that AT LEAST ONE user in the array has a website ending in .org — doesn't require all, just one or more

11. arrayItemAt — first user check

{
  "type": "arrayItemAt",
  "index": 0,
  "validations": [
    { "type": "equals", "path": "name", "value": "Leanne Graham" },
    { "type": "equals", "path": "id", "value": 1 }

}

What each validation does:

  • arrayItemAt index:0 — Targets the item at array position 0 (first item) and runs nested validations on it
  • equals path:"name" value:"Leanne Graham" — The first user's name must be "Leanne Graham"
  • equals path:"id" value:1 — The first user's id must be 1

12. String validations: startsWith, endsWith, stringLength, isNotEmpty

New string validation types for detailed text checks.

{
  "name": "user name should start with L and not be empty",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "startsWith", "path": "name", "value": "Leanne" },
    { "type": "endsWith", "path": "name", "value": "Graham" },
    { "type": "isNotEmpty", "path": "name" },
    { "type": "stringLength", "path": "name", "min": 5, "max": 50 }
  ]
}

What each validation does:

  • startsWith path:"name" value:"Leanne" — Checks that the name string begins with "Leanne"
  • endsWith path:"name" value:"Graham" — Checks that the name string ends with "Graham"
  • isNotEmpty path:"name" — Asserts the name is not an empty string, null, or undefined
  • stringLength path:"name" min:5 max:50 — Checks the string length is between 5 and 50 characters (inclusive)

13. Numeric validations: between, greaterThanOrEqual, lessThanOrEqual

Range checks and inclusive comparisons for numeric values.

{

  ]
}

What each validation does:

  • between path:"id" min:1 max:10 — Checks that id is >= 1 AND <= 10 (inclusive range)
  • greaterThanOrEqual path:"id" value:1 — Checks id >= 1
  • lessThanOrEqual path:"id" value:10 — Checks id <= 10

16. Skip test

Tests can be skipped with "skip": true. They appear as "skipped" in the report without executing.

{
  "name": "skipped test example (placeholder for future test)",
  "method": "GET",
  "endpoint": "users",
  "expectedStatus": 200,
  "skip": true
}

What each validation does:

  • skip: true — This test is skipped entirely. It won't make any HTTP request. Appears as "skipped" in the report. Useful for temporarily disabling tests or marking placeholders.

17. Custom timeout

Override the default 30s timeout with "timeout" (in milliseconds) for slow endpoints.

{
  "name": "should return users with custom timeout",
  "method": "GET",
  "endpoint": "users",
  "expectedStatus": 200,
  "timeout": 60000,
  "validations": [
    { "type": "isArray" },
    { "type": "isNotEmpty" }
  ]
}

What each validation does:

  • timeout: 60000 — Overrides the default 30-second timeout to 60 seconds for this specific test
  • isArray — Confirms the response is an array
  • isNotEmpty — Asserts the response array/string is not empty (has at least one element)

18. arrayContains / arrayNotContains

Check if an array contains (or doesn't contain) a specific value using deep equality.

{
  "name": "user tags should contain expected values",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "isArray", "path": "tags" },
    { "type": "arrayContains", "path": "tags", "value": "active" },
    { "type": "arrayNotContains", "path": "tags", "value": "banned" }
  ]
}

What each validation does:

  • isArray path:"tags" — Checks that the tags field is an array
  • arrayContains path:"tags" value:"active" — Asserts the tags array includes the value "active" (deep equality check)
  • arrayNotContains path:"tags" value:"banned" — Asserts the tags array does NOT include "banned"

19. arraySorted / arrayUnique

Validate array ordering and uniqueness. Useful for verifying API sort behavior.

{
  "name": "users should be sorted by ID and have unique IDs",
  "method": "GET",
  "endpoint": "users",
  "expectedStatus": 200,
  "validations": [
    { "type": "arraySorted", "field": "id", "order": "asc" },
    { "type": "arrayUnique", "field": "id" },
    { "type": "arrayUnique", "field": "email" }
  ]
}

What each validation does:

  • arraySorted field:"id" order:"asc" — Checks that the array is sorted by the id field in ascending order (each item's id >= previous item's id)
  • arrayUnique field:"id" — Asserts every item has a unique id value (no duplicates)
  • arrayUnique field:"email" — Asserts every item has a unique email value

20. Date validations

Validate date strings: check format, compare dates, and verify recency.

{
  "name": "order dates should be valid and recent",
  "method": "GET",
  "endpoint": "orders/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "isDate", "path": "createdAt" },
    { "type": "dateAfter", "path": "createdAt", "value": "2024-01-01" },
    { "type": "dateBefore", "path": "expiresAt", "value": "2030-12-31" },
    { "type": "dateWithinLast", "path": "updatedAt", "amount": 30, "unit": "days" }
  ]
}

What each validation does:

  • isDate path:"createdAt" — Checks that createdAt is a valid, parseable date string
  • dateAfter path:"createdAt" value:"2024-01-01" — Asserts the date is after January 1, 2024
  • dateBefore path:"expiresAt" value:"2030-12-31" — Asserts the date is before December 31, 2030
  • dateWithinLast path:"updatedAt" amount:30 unit:"days" — Asserts the date is within the last 30 days from now

21. Retry on failure

Automatically retry a test up to N times with a delay between attempts. Useful for eventual consistency or flaky endpoints.

{
  "name": "order should eventually be completed",
  "method": "GET",
  "endpoint": "orders/{{orderId}}",
  "expectedStatus": 200,
  "retry": {
    "count": 3,
    "delay": 2000
  },
  "validations": [
    { "type": "equals", "path": "status", "value": "completed" }
  ]
}

What each validation does:

  • retry count:3 delay:2000 — If validations fail, retries the entire request up to 3 more times (4 total) with a 2-second pause between attempts
  • equals path:"status" value:"completed" — Checks the status field equals "completed" — on each attempt, if this fails, the retry kicks in

Create a new post (POST)

Sends a JSON body and validates the response contains the same data with a generated id.

{

    { "type": "equals", "path": "title", "value": "Test Post Title" },

    { "type": "equals", "path": "userId", "value": 1 },
    { "type": "exists", "path": "id" }
  ]
}

What each validation does:

  • equals path:"title" value:"Test Post Title" — Confirms the created post's title matches what was sent in the request body
  • equals path:"body" value:"This is the content..." — Confirms the body text was saved correctly
  • equals path:"userId" value:1 — Confirms the userId was preserved
  • exists path:"id" — Confirms the server generated and returned a new id for the created post

CRUD Workflow with Variable Chaining 6 tests

This is the most advanced example. It demonstrates variable extraction and chaining — creating a resource, extracting its ID, then using that ID in subsequent requests.

Auto-Serial Mode: When any test in a suite uses extract, the entire suite runs in serial mode (tests execute in order). This ensures the extracted values are available for later tests.

The Workflow

  1. GET — Fetch an existing post, extract its id as postId
  2. POST — Create a new post, extract its id as newPostId
  3. GET — Verify the original post using {{postId}}
  4. PUT — Full update using {{postId}}
  5. PATCH — Partial update using {{postId}}
  6. DELETE — Delete using {{postId}}

Step 1: Fetch and Extract

The extract field captures id from the response as postId and title as originalTitle.

{
  "name": "fetch an existing post to extract its ID",
  "method": "GET",
  "endpoint": "posts/1",
  "expectedStatus": 200,
  "extract": {
    "postId": "id",
    "originalTitle": "title"
  },
  "validations": [
    { "type": "exists", "path": "id" },
    { "type": "equals", "path": "id", "value": 1 },
    { "type": "typeOf", "path": "title", "expected": "string" }
  ]
}

What each validation does:

  • extract {"postId": "id", "originalTitle": "title"} — Saves response.id into variable postId and response.title into originalTitle for use in later tests
  • exists path:"id" — Confirms the post has an id field
  • equals path:"id" value:1 — The post's id is exactly 1
  • typeOf path:"title" expected:"string" — The title field is a string type

Step 2: Create a New Post

Demonstrates POST with body and extracting the new post's ID as newPostId.

{
  "name": "create a new post using extracted userId reference",
  "method": "POST",
  "endpoint": "posts",
  "body": {
    "title": "My Automated Test Post",
    "body": "This post was created by the API automation framework.",
    "userId": 1
  },
  "expectedStatus": 201,
  "extract": { "newPostId": "id", "postTitle": "title" },
  "validations": [
    { "type": "exists", "path": "id" },
    { "type": "equals", "path": "title", "value": "My Automated Test Post" }
  ]
}

What each validation does:

  • extract {"newPostId": "id", "postTitle": "title"} — Saves the newly created post's id as newPostId and its title as postTitle
  • exists path:"id" — Server returned a generated id for the new post
  • equals path:"title" value:"My Automated Test Post" — The created post has the correct title

Step 3: Use Extracted Variable

The {{postId}} placeholder in the endpoint is replaced with the value extracted in Step 1.

{
  "name": "verify original post still exists using chained variable",
  "method": "GET",
  "endpoint": "posts/{{postId}}",
  "expectedStatus": 200,
  "validations": [
    { "type": "equals", "path": "id", "value": 1 }
  ]
}

Conditional Validations (if/then/else)

Run different validations depending on response content. The condition is any validation type — if it passes, the then branch runs; otherwise the else branch runs.

Conditional based on response field

If the user has a company, validate company fields exist; otherwise ensure they're absent.

{
  "name": "conditional company validation",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "exists", "path": "name" },
    {
      "type": "if",
      "condition": { "type": "exists", "path": "company" },
      "then": [
        { "type": "exists", "path": "company.name" },
        { "type": "exists", "path": "company.catchPhrase" }
      ],
      "else": [
        { "type": "notExists", "path": "company.name" }
      ]
    }
  ]
}

Since JSONPlaceholder users always have a company object, the then branch executes and validates company sub-fields.

What each validation does:

  • exists path:"name" — First, confirm the name field exists (basic sanity check)
  • if condition:{exists path:"company"} — Evaluate: does the company field exist?
  • then: [exists "company.name", exists "company.catchPhrase"] — YES path: validate that company sub-fields exist
  • else: [notExists "company.name"] — NO path: confirm company sub-fields are absent. Since JSONPlaceholder always has company, the then branch runs.

Conditional without else branch

If the user has a website, check it contains a dot. If no website, skip silently.

{
  "name": "optional website validation",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    {
      "type": "if",
      "condition": { "type": "exists", "path": "website" },
      "then": [
        { "type": "contains", "path": "website", "value": "." }
      ]
    }
  ]
}

What each validation does:

  • if condition:{exists path:"website"} — Check if the user has a website field
  • then: [contains path:"website" value:"."] — If yes: verify the website contains a dot (basic URL check). If no website field exists, nothing happens — the validation passes silently.
