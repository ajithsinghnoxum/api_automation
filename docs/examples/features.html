Feature Examples

Comprehensive examples of all framework features — built-in variables, data-driven tests, hooks, polling, and more.

Built-in Variables

Use dynamic built-in variables in endpoints, body, and query params. These generate fresh values on each test run.

Create a unique resource with generated values

Uses {{$randomEmail}}, {{$randomString}}, and {{$isoDate}} to create unique test data every run.

{
  "name": "create user with unique data",
  "method": "POST",
  "endpoint": "users",
  "body": {
    "name": "User {{$randomString}}",
    "email": "{{$randomEmail}}",
    "createdAt": "{{$isoDate}}"
  },
  "expectedStatus": 201,
  "validations": [
    { "type": "exists", "path": "id" }
  ]
}

What each part does:

  • body "name": "User {{$randomString}}" — Generates a name like "User a3f9b2c1d4e5" with a random 12-char string appended each run
  • body "email": "{{$randomEmail}}" — Generates a unique email like "test42857@example.com" to avoid duplicate conflicts
  • body "createdAt": "{{$isoDate}}" — Inserts the current ISO timestamp (e.g., "2026-03-11T10:30:00.000Z")
  • exists path:"id" — Confirms the server generated and returned an id for the created user

Available Built-in Variables

VariableOutput
{{$timestamp}}Unix timestamp (e.g., 1710000000000)
{{$isoDate}}ISO date (e.g., 2024-03-10T12:00:00Z)
{{$guid}} / {{$uuid}}Random UUID v4
{{$randomInt}}Random int 1–100000
{{$randomEmail}}Random email like test42@example.com
{{$randomString}}12-char alphanumeric string

Data-Driven Tests 3 tests

Run the same test multiple times with different data. Each dataSet entry creates a separate test labeled [data #1], [data #2], etc.

Parameterized user lookup

Fetches 3 different users by ID and validates each has the correct name.

{
  "name": "should return correct user by ID",
  "method": "GET",
  "endpoint": "users/{{id}}",
  "expectedStatus": 200,
  "dataSet": [
    { "id": 1, "expectedName": "Leanne Graham" },
    { "id": 2, "expectedName": "Ervin Howell" },
    { "id": 3, "expectedName": "Clementine Bauch" }
  ],
  "validations": [
    { "type": "equals", "path": "name", "value": "{{expectedName}}" }
  ]
}

What each part does:

  • endpoint: "users/{{id}}" — The {{id}} placeholder is replaced with each dataset's id value (1, 2, 3), creating 3 separate test runs
  • dataSet — Three entries: each generates its own test labeled [data #1], [data #2], [data #3]
  • equals path:"name" value:"{{expectedName}}" — For each dataset row, checks the response name matches the expected name. E.g., user 1 should be "Leanne Graham"

Parameterized post filtering

Tests filtering by different user IDs and validates results.

{
  "name": "should filter posts by userId",
  "method": "GET",
  "endpoint": "posts",
  "queryParams": { "userId": "{{userId}}" },
  "expectedStatus": 200,
  "dataSet": [
    { "userId": 1, "minPosts": 5 },
    { "userId": 2, "minPosts": 5 }
  ],
  "validations": [
    { "type": "isArray" },
    { "type": "arrayLength", "min": "{{minPosts}}" },
    {
      "type": "arrayEvery",
      "validations": [
        { "type": "equals", "path": "userId", "value": "{{userId}}" }
      ]
    }
  ]
}

What each part does:

  • queryParams: { "userId": "{{userId}}" } — Filters posts by userId from each dataset row
  • dataSet — Two entries: tests filtering for userId 1 and userId 2
  • isArray — Response is an array
  • arrayLength min: "{{minPosts}}" — Ensures at least minPosts results (5 for both datasets)
  • arrayEvery > equals path:"userId" value:"{{userId}}" — Every returned post belongs to the filtered user — no cross-user results

Tagged Tests

Tags categorize tests for filtering in the Web UI. Click tag chips to quickly select tests by category.

Using tags for test categorization

[
  {
    "name": "get all users",
    "method": "GET",
    "endpoint": "users",
    "expectedStatus": 200,
    "tags": ["smoke", "users"],
    "validations": [{ "type": "isArray" }]
  },
  {
    "name": "create a post",
    "method": "POST",
    "endpoint": "posts",
    "expectedStatus": 201,
    "tags": ["regression", "posts"],
    "body": { "title": "Test", "body": "Content", "userId": 1 }
  },
  {
    "name": "get user by ID",
    "method": "GET",
    "endpoint": "users/1",
    "expectedStatus": 200,
    "tags": ["smoke", "users"],
    "validations": [{ "type": "equals", "path": "id", "value": 1 }]
  }
]

What each part does:

  • tags: ["smoke", "users"] — Categorizes this test under "smoke" and "users" tags. In the web UI, clicking the "smoke" tag chip shows only tests with that tag. Via CLI: TEST_TAGS=smoke npx playwright test
  • tags: ["regression", "posts"] — This POST test is tagged for regression testing only
  • Combined: You can filter by multiple tags and the UI auto-hides suites with no matching tests

In this example, clicking the "smoke" filter chip shows only the GET tests. Clicking "regression" shows only the POST test.

Retry Configuration

Retry failed tests automatically with configurable count and delay. Useful for eventual consistency scenarios.

Retry with delay

Retries the request up to 3 more times (4 total attempts) with 2 seconds between each attempt.

{
  "name": "order should be processed",
  "method": "GET",
  "endpoint": "orders/{{orderId}}",
  "expectedStatus": 200,
  "retry": {
    "count": 3,
    "delay": 2000
  },
  "validations": [
    { "type": "equals", "path": "status", "value": "completed" }
  ]
}

What each part does:

  • retry count:3 delay:2000 — On failure, waits 2 seconds then retries. Up to 3 retries (4 total attempts). If ANY attempt passes, the test passes.
  • equals path:"status" value:"completed" — The validation being retried — checks if the order status has changed to "completed". Useful when an async process needs time to finish.

If the first attempt fails (e.g., status is still "pending"), the framework waits 2 seconds and tries again. If any attempt passes, the test passes. If all 4 attempts fail, the error from the last attempt is reported.

Conditional Validations (if/then/else)

Run different validations depending on response content. The condition is any validation type — if it passes, the then branch runs; otherwise the else branch runs.

Conditional based on response field

If the user has a company, validate company fields exist; otherwise ensure they're absent.

{
  "name": "conditional company validation",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    { "type": "exists", "path": "name" },
    {
      "type": "if",
      "condition": { "type": "exists", "path": "company" },
      "then": [
        { "type": "exists", "path": "company.name" },
        { "type": "exists", "path": "company.catchPhrase" }
      ],
      "else": [
        { "type": "notExists", "path": "company.name" }
      ]
    }
  ]
}

What each part does:

  • exists path:"name" — Basic sanity check that the response has a name field
  • if condition:{exists path:"company"} — Evaluates whether the company field exists in the response
  • then — If company EXISTS: checks company.name and company.catchPhrase are present
  • else — If company is MISSING: confirms company.name does not exist. The framework picks exactly one branch to execute.

Since JSONPlaceholder users always have a company object, the then branch executes and validates company sub-fields.

Conditional without else branch

If the user has a website, check it contains a dot. If no website, skip silently.

{
  "name": "optional website validation",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "validations": [
    {
      "type": "if",
      "condition": { "type": "exists", "path": "website" },
      "then": [
        { "type": "contains", "path": "website", "value": "." }
      ]
    }
  ]
}

What each part does:

  • if condition:{exists path:"website"} — Checks if the user has a website field
  • then: [contains path:"website" value:"."] — Only runs if website exists: verifies it contains a dot. If no website, this validation is silently skipped (no error, no else branch needed).

Pre/Post Request Hooks

Use beforeRequest to modify the request before it's sent, and afterResponse to extract data or set variables after receiving the response.

beforeRequest — Add dynamic headers

Inject a timestamp header before the request is sent:

{
  "name": "get users with request ID",
  "method": "GET",
  "endpoint": "users",
  "expectedStatus": 200,
  "beforeRequest": "request.headers = request.headers || {}; request.headers['X-Request-Time'] = new Date().toISOString();",
  "validations": [
    { "type": "isArray" },
    { "type": "arrayLength", "min": 1 }
  ]
}

What each part does:

  • beforeRequest: "request.headers = ..." — JavaScript code that runs BEFORE the HTTP request is sent. Here it adds a custom X-Request-Time header with the current timestamp. The hook has access to request (modifiable), vars, and env.
  • isArray — After the request completes, validates the response is an array
  • arrayLength min:1 — The array has at least 1 item

afterResponse — Extract and store variables

Save response data into the shared variable store for use in later tests:

{
  "name": "get first user",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "afterResponse": "vars.userName = response.data.name; vars.userEmail = response.data.email;",
  "validations": [
    { "type": "exists", "path": "name" }
  ]
}

What each part does:

  • afterResponse: "vars.userName = response.data.name; ..." — JavaScript code that runs AFTER the response is received. Saves name and email from the response into the shared variable store. Later tests can use {{userName}} and {{userEmail}}.
  • exists path:"name" — Standard validation that the name field exists in the response

The hook context provides: request (method, endpoint, headers, body), response (status, data), vars (shared variable store), env (process.env).

Poll Mode

Poll mode repeats a request at intervals until a condition passes or a timeout is reached. Ideal for waiting on async operations.

Wait for a resource to have expected data

This example polls a user endpoint every 2 seconds for up to 10 seconds, waiting for the name field to exist:

{
  "name": "wait for user data",
  "method": "GET",
  "endpoint": "users/1",
  "expectedStatus": 200,
  "poll": {
    "interval": 2000,
    "timeout": 10000,
    "condition": { "type": "exists", "path": "name" }
  },
  "validations": [
    { "type": "exists", "path": "name" },
    { "type": "typeOf", "path": "name", "expected": "string" },
    { "type": "exists", "path": "email" }
  ]
}

What each part does:

  • poll interval:2000 — Re-sends the request every 2 seconds
  • poll timeout:10000 — Gives up after 10 seconds total
  • poll condition:{exists path:"name"} — The "stop polling" condition — when name exists in the response, polling stops and full validations run
  • exists path:"name" — Full validation: name field must be present
  • typeOf path:"name" expected:"string" — Full validation: name must be a string type
  • exists path:"email" — Full validation: email field must be present. All three must pass after the poll condition is met.

Poll mode activates only if the initial request fails validations. The condition is any validation type — when it passes, all validations are re-checked.

Conditional Execution (onlyIf)

Skip tests based on environment or variable values using the onlyIf field.

Environment-based execution

Only run this test when TEST_ENV=staging:

{
  "name": "staging-only cleanup",
  "method": "DELETE",
  "endpoint": "test-data/reset",
  "expectedStatus": 200,
  "onlyIf": { "env": "staging" }
}

What each part does:

  • onlyIf: { "env": "staging" } — This test is ONLY executed when TEST_ENV=staging. In any other environment (or if TEST_ENV is not set), the test is automatically skipped and shows as "skipped" in the report.

Run with: TEST_ENV=staging npx playwright test

Variable-based execution

Only run if a previously extracted variable matches a value:

{
  "name": "verify premium user features",
  "method": "GET",
  "endpoint": "users/{{userId}}/premium",
  "expectedStatus": 200,
  "onlyIf": { "var": "userRole", "equals": "premium" },
  "validations": [
    { "type": "exists", "path": "premiumFeatures" }
  ]
}

What each part does:

  • onlyIf: { "var": "userRole", "equals": "premium" } — Only runs if a previously extracted variable named userRole has the value "premium". If the variable doesn't exist or has a different value, the test is skipped.
  • exists path:"premiumFeatures" — If the test runs, validates the premium features field exists

If equals is omitted, the test runs when the variable is truthy (any non-empty/non-null value).

Shared Fixtures ($ref)

Define reusable data in fixtures.json and reference it with $ref in test bodies or anywhere in the test config.

Project-level fixtures.json

// test-configs/jsonplaceholder/fixtures.json
{
  "newPost": {
    "title": "Test Post",
    "body": "This is a test post body",
    "userId": 1
  },
  "updatedPost": {
    "title": "Updated Title",
    "body": "Updated body content",
    "userId": 1
  }
}

Using $ref in test cases

{
  "suite": "Post CRUD with Fixtures",
  "tests": [
    {
      "name": "create post using fixture",
      "method": "POST",
      "endpoint": "posts",
      "expectedStatus": 201,
      "body": { "$ref": "fixtures.newPost" },
      "validations": [
        { "type": "equals", "path": "title", "value": "Test Post" },
        { "type": "exists", "path": "id" }
      ]
    },
    {
      "name": "update post using fixture",
      "method": "PUT",
      "endpoint": "posts/1",
      "expectedStatus": 200,
      "body": { "$ref": "fixtures.updatedPost" },
      "validations": [
        { "type": "equals", "path": "title", "value": "Updated Title" }
      ]
    }
  ]
}

What each part does:

  • body: { "$ref": "fixtures.newPost" } — At runtime, this is replaced with the full newPost object from fixtures.json: { "title": "Test Post", "body": "...", "userId": 1 }. No need to repeat the data inline.
  • equals path:"title" value:"Test Post" — Validates the response title matches what the fixture sent
  • exists path:"id" — Server generated an id for the new post
  • body: { "$ref": "fixtures.updatedPost" } — Uses the updatedPost fixture for the PUT request body
  • equals path:"title" value:"Updated Title" — Validates the update used the fixture data correctly

Suite-level fixtures can also be defined inline and take precedence over the shared fixtures.json. Nested paths are supported: "$ref": "fixtures.users.admin".

OpenAPI / Swagger Import

Automatically generate test suites from an OpenAPI 3.x or Swagger 2.0 spec file.

How to import

  • Click "Import OpenAPI" in the web UI sidebar
  • Select your OpenAPI/Swagger JSON file
  • A new project is created with suites grouped by API tags
  • Each endpoint becomes a test case with auto-generated validations

What gets generated:

  • Base URL from servers[0].url (v3) or host + basePath (v2)
  • Request bodies from schema definitions with example values
  • exists + typeOf validations for required response fields
  • isArray + arrayEvery for array response schemas
  • Path params converted to {{variable}} syntax
  • Query params from parameter definitions

PDF Export

Export any test run as a clean, printable PDF report.

Export options

  • From Run History: click the picture_as_pdf icon next to any run
  • From Run Detail: click the PDF button in the run detail panel
  • Via API: GET /api/projects/{projectId}/runs/{runId}/pdf

The PDF includes a summary dashboard with pass/fail/skip counts, suite-grouped test tables with durations, and color-coded status indicators. Use your browser's Print dialog (Ctrl+P) to save as PDF.

CSV & JUnit Export

After running tests, export results for external tools.

Export options

From the Run History view in the web UI:

  • JUnit XML — click the download icon for CI/CD integration (Jenkins, GitHub Actions, Azure DevOps)
  • CSV — click the table_chart icon for spreadsheet analysis (Excel, Google Sheets)
  • Trend Chart — visible at the top of Run History showing pass/fail trends across the last 20 runs

Or use the API directly:

GET /api/projects/{projectId}/runs/{runId}/junit   → application/xml
GET /api/projects/{projectId}/runs/{runId}/csv     → text/csv
GET /api/projects/{projectId}/trends               → application/json

Try & Auto-Generate Validations

Skip writing validations manually — let the framework learn from a real API response. The "Try & Auto-Generate" button in the test editor sends the request and auto-generates validations from the actual response data.

Step-by-step workflow

  1. Open a test case and set the HTTP method and endpoint (e.g., GET + users/1)
  2. Optionally check "Include value checks" in the modal footer to also generate equals for every value
  3. Click "Try & Auto-Generate" — the server fires the real request with your project's auth
  4. Review the populated validations, tweak as needed, and Save

Example: Object response (without value checks)

Given an endpoint that returns:

{
  "meta": { "id": 2683400, "guid": "3d70fcc0-...", "typeRef": 2503539 },
  "values": [
    { "attribute": 998, "value": false },
    { "attribute": 1012, "language": 201, "value": "aasa" }
  ]
}

Auto-generated validations (structural only):

"validations": [
  { "type": "schema", "properties": { "meta": "object", "values": "array" } },
  { "type": "exists", "path": "meta" },
  { "type": "typeOf", "path": "meta", "expected": "object" },
  { "type": "exists", "path": "values" },
  { "type": "isArray", "path": "values" },
  { "type": "arrayLength", "path": "values", "exact": 2 }
]

What each validation does:

  • schema — Validates top-level structure: meta must be an object and values must be an array
  • exists meta — Confirms the meta field is present in the response
  • typeOf meta "object" — Asserts meta is a JavaScript object (not null, not array)
  • isArray values — Confirms values is an array (uses Array.isArray(), not typeof)
  • arrayLength values exact: 2 — Asserts the array has exactly 2 items (adjust to min: 1 for flexibility)

Example: With "Include value checks" enabled

When the checkbox is checked, deep equals validations are added for every primitive value, recursing into nested objects and array items:

"validations": [
  // ... structural validations (schema, exists, typeOf, isArray) ...

  // Deep value checks generated automatically:
  { "type": "equals", "path": "meta.id", "value": 2683400 },
  { "type": "equals", "path": "meta.guid", "value": "3d70fcc0-..." },
  { "type": "equals", "path": "meta.typeRef", "value": 2503539 },
  { "type": "equals", "path": "values[0].attribute", "value": 998 },
  { "type": "equals", "path": "values[0].value", "value": false },
  { "type": "equals", "path": "values[1].attribute", "value": 1012 },
  { "type": "equals", "path": "values[1].language", "value": 201 },
  { "type": "equals", "path": "values[1].value", "value": "aasa" }
]

When to use value checks:

  • Snapshot testing — Capture the current response as a baseline; any future change is detected immediately
  • Static data endpoints — Perfect for config, metadata, or lookup APIs where values don't change
  • Refine after generation — Remove volatile fields (timestamps, random IDs) and keep the stable ones

Example: Array response

For a GET /users that returns [{ "id": 1, "name": "Alice" }, ...]:

"validations": [
  { "type": "isArray" },
  { "type": "arrayLength", "exact": 10 },
  { "type": "schema", "path": "[0]", "properties": { "id": "number", "name": "string" } },
  { "type": "arrayEvery", "validations": [
    { "type": "typeOf", "path": "id", "expected": "number" },
    { "type": "typeOf", "path": "name", "expected": "string" }
  ]}
]

What each validation does:

  • isArray (no path) — Confirms the entire response body is an array
  • arrayLength exact: 10 — The array has exactly 10 items
  • schema [0] — First item has id (number) and name (string)
  • arrayEvery — Ensures every item in the array has a numeric id and string name
Server endpoint: POST /api/projects/:id/try-request — accepts method, endpoint, queryParams, body, and includeValues. Returns the response data plus auto-generated validations.

Chain Builder / Flow Editor

Visualize how tests in a suite pass data to each other. The Chain Builder auto-detects dependencies from extract and {{variable}} usage and renders a connected flow diagram.

Example: Login → Create → Verify chain

A typical 3-step workflow where each test feeds data to the next:

[
  {
    "name": "Login",
    "method": "POST",
    "endpoint": "auth/login",
    "body": { "username": "admin", "password": "secret" },
    "expectedStatus": 200,
    "extract": { "token": "data.accessToken", "userId": "data.user.id" }
  },
  {
    "name": "Create Post",
    "method": "POST",
    "endpoint": "users/{{userId}}/posts",
    "headers": { "Authorization": "Bearer {{token}}" },
    "body": { "title": "Hello World" },
    "expectedStatus": 201,
    "extract": { "postId": "data.id" }
  },
  {
    "name": "Verify Post",
    "method": "GET",
    "endpoint": "posts/{{postId}}",
    "headers": { "Authorization": "Bearer {{token}}" },
    "expectedStatus": 200,
    "validations": [
      { "type": "equals", "path": "title", "value": "Hello World" }
    ]
  }
]

What the Chain Builder shows:

  • Login node with green badges: token, userId (extracted)
  • Create Post node with blue badges: token, userId (consumed) and green badge: postId (extracted)
  • Verify Post node with blue badges: postId, token (consumed)
  • Purple connector arrows from Login → Create Post (token, userId) and from Create Post → Verify Post (postId), plus Login → Verify Post (token)

Step-Through Execution

Click "Step Through" in the chain view to execute tests one at a time:

  1. The first test (Login) is highlighted as active
  2. Click Next — the real API request is sent and the response is shown inline
  3. Extracted variables are stored and available for subsequent tests
  4. Each completed test is marked passed or failed
  5. Use Previous to review earlier responses, or Stop to exit
Tip: If a test uses a {{variable}} that no previous test extracts, the chain view displays a warning. Fix these before running the chain to avoid failures.

Test Templates / CRUD Generator

Generate a complete test suite from a template instead of creating tests one by one. Click "From Template" in the project header.

Example: CRUD Suite for "products"

Fill in: Resource name = products, Singular name = product, ID field = id

The template generates 6 tests with variable chaining:

{
  "suite": "Products CRUD",
  "tests": [
    { "name": "List all products",    "method": "GET",    "endpoint": "products",                "expectedStatus": 200 },
    { "name": "Create product",       "method": "POST",   "endpoint": "products",                "expectedStatus": 201,
      "extract": { "productId": "id" } },
    { "name": "Get product by ID",    "method": "GET",    "endpoint": "products/{{productId}}", "expectedStatus": 200 },
    { "name": "Update product",       "method": "PUT",    "endpoint": "products/{{productId}}", "expectedStatus": 200 },
    { "name": "Delete product",       "method": "DELETE", "endpoint": "products/{{productId}}", "expectedStatus": 200 },
    { "name": "Verify product deleted","method": "GET",    "endpoint": "products/{{productId}}", "expectedStatus": 404 }
  ]
}

Available templates:

  • CRUD Suite — Full resource lifecycle (list, create, get, update, delete, verify)
  • Auth Flow — Register, login, protected access, unauthorized check, logout
  • Pagination Check — First page, second page, large page, custom limit
  • Health & Status — Health check, response time, version endpoint
  • Search & Filter — Keyword search, no results, sort asc/desc
  • Error Handling — 404, 400, invalid path, 405 responses
Workflow tip: Generate a CRUD suite from the template, then use "Try & Auto-Generate" on each test to add real validations from live API responses.

Bulk Test Operations

Perform actions across multiple tests at once without editing them one by one.

How to use Bulk Edit mode

  1. Click "Bulk Edit" in the project header (button highlights when active)
  2. Checkboxes appear next to each test — click to select individual tests, or use the suite checkbox to select all tests in a suite
  3. A toolbar appears showing the number of selected tests and available actions
  4. Click an action button (e.g., Add Tags, Skip, Auto-Generate)
  5. Click "Bulk Edit" again to exit bulk mode

Common bulk workflows

  • Tag all smoke tests: Select tests → Add Tags → type smoke
  • Skip flaky tests: Select tests → Skip → enter flaky, investigating
  • Migrate endpoints: Select all → Find/Replace → find v1 replace with v2
  • Re-validate after deploy: Select all → Auto-Generate → rebuilds validations from live responses
  • Clean up: Select obsolete tests → Delete

Response Comparison / Diff

Compare actual API response bodies between two test runs to detect unintended changes.

Snapshot & Compare workflow

  1. Open History in the project view
  2. Click "Snapshot" to capture all current API responses
  3. Make your changes (deploy, update config, etc.)
  4. Click "Snapshot" again to capture the new state
  5. Check both snapshots and click "Compare"
  6. Review the diff — filter by changed, added, or removed
  7. Expand any changed test to see a side-by-side line diff of the response body

What the diff shows

  • Green lines — Added in the newer run
  • Red lines — Removed from the older run
  • Unmarked lines — Unchanged between runs
  • Filter buttons let you focus on just the changes
Use case: Take a snapshot before and after a deployment to catch unintended API response changes — new fields, missing fields, changed values, or different status codes.

Schema Drift Detection

Detect when API response structures change from a known-good baseline — field additions, removals, and type changes.

Baseline & Detect workflow

  1. Open History and click "Save Baseline" — captures the schema (field names + types) for every test
  2. Deploy or make API changes
  3. Click "Detect Drift" — re-fetches all endpoints and compares against the baseline
  4. The drift report shows a color-coded progress bar and per-test results
  5. Expand any drifted test to see field-level changes

What the drift report shows

  • Field added — New field in the response (e.g., user.avatar appeared)
  • Field removed — Missing field (e.g., user.legacyId gone)
  • Type changed — Field type changed (e.g., count: string → number)
  • Progress bar shows stable (green) vs drifted (red) ratio at a glance
Best practice: Save a baseline when your API is in a known-good state. Run drift detection after each deployment to catch unintended structural changes before they reach users.

Scheduled Test Runs

Automatically run your tests at regular intervals without manual triggering.

Setting up a schedule

  1. Click "Schedules" in the project header
  2. Enter a name (e.g., "Nightly Smoke Tests")
  3. Select a preset like "Daily at 9:00 AM" or enter a custom cron: 0 9 * * 1-5
  4. Click "Add" — the schedule starts immediately
  5. Results appear in History tagged as [scheduled:name]

Common scheduling patterns

  • */5 * * * * — Every 5 minutes (continuous monitoring / smoke tests)
  • 0 * * * * — Every hour on the hour
  • 0 0 * * * — Daily at midnight (nightly regression)
  • 0 9 * * 1-5 — Weekdays at 9 AM (business hours only)
  • 0 */6 * * * — Every 6 hours (4 times per day)
Tip: Pause/resume schedules without deleting them. Combine with notifications for hands-free monitoring.

KV Editors & Per-Test Headers

Query parameters and per-test headers use structured key-value editors. Per-test headers are merged with project auth headers at runtime.

Test with per-test headers

Define custom headers that are sent alongside your project's auth headers.

{
  "name": "create resource with custom headers",
  "method": "POST",
  "endpoint": "resources",
  "headers": {
    "X-Request-ID": "{{$guid}}",
    "X-Custom-Source": "api-tests",
    "Accept-Language": "en-US"
  },
  "body": { "name": "Test Resource" },
  "expectedStatus": 201,
  "validations": [
    { "type": "exists", "path": "id" }
  ]
}
  • headers — Key-value pairs added to the request. Merged with project-level auth headers (Bearer, Basic, API Key)
  • {{$guid}} — Built-in variable resolved at runtime, generating a unique UUID for each request
  • Headers support all variable types: built-in ($timestamp), extracted ({{token}}), and dataset variables
  • Per-test headers are included when using Copy as cURL

Query params via KV editor

Instead of typing ?page=1&limit=10, use the visual KV editor in the test modal.

{
  "name": "list users with pagination",
  "method": "GET",
  "endpoint": "users",
  "queryParams": {
    "page": "1",
    "limit": "10",
    "sort": "name",
    "order": "asc"
  },
  "expectedStatus": 200
}
  • Each key-value pair is a separate row in the editor with its own input fields
  • Click "Add Param" to add rows, click × to remove
  • Query params also support {{variable}} syntax
